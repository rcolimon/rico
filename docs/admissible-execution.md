# Admissible Execution

## Definition

Admissible execution is execution that remains legitimate under current runtime conditions.

Execution is not admissible simply because it began correctly.

It must remain justified throughout its lifecycle.

---

## Execution vs Admissibility

Execution asks:

> Can the system continue?

Admissibility asks:

> Does the system still possess a valid basis to continue?

A technically functioning system may no longer be admissible.

---

## Conditions for Admissibility

Execution remains admissible only while:

- authority remains valid,
- context remains applicable,
- evaluability is preserved,
- governance constraints remain satisfied,
- anticipated consequences remain acceptable.

If any of these conditions degrade, admissibility must be reassessed.

---

## Runtime Reassessment

Admissibility is continuously re-evaluated as:

- system state changes,
- external conditions evolve,
- authority is modified,
- observations update,
- new consequences emerge.

Runtime governance exists to perform this reassessment.

---

## Possible Outcomes

When admissibility changes, governance may:

- Continue execution
- Constrain execution
- Escalate for human review
- Suspend execution
- Terminate execution

The response depends on present conditions, not historical authorization.

---

## Architectural Principle

Execution continuity and execution admissibility are independent properties.

A system may continue executing correctly while no longer remaining admissible.

Runtime governance exists to detect that distinction.

---

**Humans First. Continuity Always. Architecture Must Survive.**
